Multi- Agent AI Solution for Payment System Development:

The diagram illustrates an Multi-Agent architecture powered by CrewAI, orchestrating multiple agents to automate the development of a payment application using the Sikka API v4 catalog.

Feature requirements are first extracted from a reference solution like Nadapayments through web search powered by Exa API.

These features are then semantically matched to relevant Sikka APIs using FAISS and retrieval-augmented generation (RAG) techniques.

API document links are processed to extract input/output parameters using headless browsing tools such as Playwright.

Based on this information, full-stack application code is generated using Flask for the backend and React for the frontend, guided by outputs from previous agents. The final system is reviewed for correctness, completeness, and production-readiness, ensuring alignment with the assignment objectives.

Data Flow Summary:

User Input: Reference_Solution_URL + Sikka CSV

Research Agent features

API Retriever Agent feature-to-API table

Parameter Extractor Agent Input/Output Parameters

Developer Agent code files

Reviewer Agent QA summary

Final Output: Full-stack application + QA validation

Tech Stack Used

FAISS (Facebook AI Similarity Search): Used for efficient similarity search and retrieval of relevant Sikka APIs from embedded documentation.

Sentence Transformers (all-MiniLM-L6-v2): Used to generate embeddings of API documentation content for similarity-based search.

Exa Search API: Utilized for web-based document search and preview to simulate research and extract payment platform features.

CrewAI Framework: Manages a multi-agent system with sequential task execution for research, API retrieval, development, and review.

OpenAI GPT-3.5: Powers all agents as the underlying language model for reasoning, planning, and generating responses or code.

Defined a custom wrapper around the GPT-3.5-turbo model to standardize prompt formatting making the model callable like a function across agents. This ensures consistent interaction by wrapping inputs into a system-user message format required by OpenAI's ChatCompletion API.

# Custom GPT wrapper
class GPTWrapper:
    def __init__(self, model="gpt-3.5-turbo"):
        self.model = model
    def __call__(self, prompt):
        if isinstance(prompt, dict) and "task" in prompt:
            prompt = prompt["task"]
        elif not isinstance(prompt, str):
            prompt = str(prompt)
        response = openai.ChatCompletion.create(
            model=self.model,
            messages=[{"role": "system", "content": "You are a helpful assistant."},
                     {"role": "user", "content": prompt}]
        )
        return response.choices[0].message["content"]

# FAISS Tool
class SikkaAPIRetrieverToolSchema(BaseModel):
    query: str

class SikkaAPIRetrieverTool(BaseTool):
    name: str = Field(default="SikkaAPI")
    description: str = Field(default="Retrieve relevant Sikka API endpoints from embedded knowledge base.")
    args_schema: Type[BaseModel] = SikkaAPIRetrieverToolSchema

    def _run(self, query: str) -> Any:
        index = faiss.read_index("sikka_faiss.index")
        docs = pickle.load(open("sikka_docs.pkl", "rb"))
        metadata = pickle.load(open("sikka_meta.pkl", "rb"))
        model = SentenceTransformer("all-MiniLM-L6-v2")
        query_embedding = model.encode([query])
        D, I = index.search(query_embedding, 5)
        return "\n\n".join([f"{metadata[i]['api_name']} - {metadata[i]['endpoint']}\n{docs[i]}" for i in I[0]])

# Exa Tool
class ExaParamExtractorSchema(BaseModel):
    query: str

class ExaParamExtractorTool(BaseTool):
    name: str = Field(default="WebResearcher")
    description: str = Field(default="Extract key features or documentation parameters from the web using a search query.")
    args_schema: Type[BaseModel] = ExaParamExtractorSchema

    def _run(self, query: str) -> Any:
        exa = Exa(api_key=os.environ["EXA_API_KEY"])
        results = exa.search_and_contents(query, use_autoprompt=True, num_results=3)
        if not results.results:
            return "No useful content found."
        return "\n\n".join([f"{r.title} - {r.url}\n{r.text[:1000]}" for r in results.results])

Parameter Extraction Agent

We can add a Parameter Extraction Agent to scrape required and optional parameters from API docs link (API retriver Agent output as below) using tools like Playwright, enabling replacement of current placeholder inputs with accurate schemas.

While Exa effectively retrieves summarized documentation content, it does not reliably return structured parameter schemas (e.g., JSON fields, types, request/response examples). Exa Cannot parse url's with dynamic elements (tabs, toggles) or "#" in them.

Why was this not implemented:

The Sikka API documentation is embedded in JavaScript-heavy web pages, which cannot be parsed reliably using basic HTML scrapers like BeautifulSoup or standard web search tools like Exa.

Explored tools such as Exa, requests-HTML, and BeautifulSoup, but they failed to render or access the full content of the documentation links.

Headless browser tools like Playwright or Selenium were identified as suitable alternatives; however, they require a runtime environment with GUI or async browser support, which is not feasible in current notebook platforms like Kaggle. Did not implement this because of time restrictions.

Crew & Process :

Kick off a sequential CrewAI pipeline where each agent passes output as context to the next, simulating an Agentic RAG-driven app builder.

Crew is instantiated by combining all defined agents and tasks into a single executable pipeline.

Process.sequential ensures that agents execute one after another, in the exact order:

Research Agent API Retriever Developer Reviewer.

https://docs.crewai.com/introduction

https://docs.crewai.com/concepts/crews

Process.sequential is the best fit because each agents task depends on the output of the previous agent (e.g., API retrieval relies on extracted features).

The flow mimics a real-world development pipeline: research design implementation review.

crew.kickoff() triggers the entire agent chain and coordinates their execution
